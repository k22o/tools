# AI Coding

## AIを使ったコーディング

### スペック駆動開発

- スペック駆動開発とは: 仕様書を信頼できる情報源として、AIに実装させる
   - バイブコーディングは「いきあたりばったり」。低品質・保守性の低いものができがち
   - 事前に設計を仕様書として落とし込むことで、高品質な結果を得られる

### スペック駆動開発のポイント

- 永続ドキュメントとステアリングドキュメントを併用する
   - 永続ドキュメント: 要求定義書・昨日設計書・技術ガイドライン etc
   - ステアリングドキュメント: 各施行での具体的な実装タスク
   - (参考) https://github.com/GenerativeAgents/claude-code-book-chapter4/blob/main/CLAUDE.md


## コーディングエージェント活用時のトラブルと対処法

- 過去の指示を忘れてしまう
   - context windowの容量制限
   - context rotによる性能低下 (contextが多いほど性能が落ちる)
   - => context エンジニアリング
- 意図した通りに動かない
   - 暗黙のルールが伝わっていない
   - 資料間で矛盾がある
   - => 指示追従性を高める
- 外部システムと連携できない
   - => MCPの利用

### context エンジニアリング

contextをうまく制御するための方法

- 圧縮: contextの情報を圧縮する
- 構造化ノート: 重要な情報をファイルに吐き出して永続化する
  - 「タスクリストを作成してファイルに出力、タスクが終わったらチェックマークを入れてください」
  - 「発生している問題は、problem.mdに出力して」
  - 「各フェーズの完了時に、自動的にtasklist.mdを更新して」
- サブエージェントの利用: 親Agentから独立した、別のcontextを持つagent
  - 分業されたタスクをこなす
  - コンテキストを圧迫しがちなレビューやテストなどをsub-agentにするのがよい

### 指示追従性を高める

(例)

- スキルを使って、必要に応じて特定の知識を利用する
  - 規定されたプロンプトにそった処理を実行できる
  - すべてが1つのmdファイルにあると、contextが増えるので、別ファイルに分割する
- hookを使って、明示的にやるべきことを設定する 


## プロンプト作成のポイント

### 記載する情報

「意図」「コンテンツ」「コンテクスト」を明記する

- 意図: 何をさせたいか
  - 例: ユーザー情報を格納するクラスを作成したい
- コンテンツ: 何を作るか・どんな条件か (WHAT)
  - 例: メンバー変数、メソッド、コンストラクタの情報
- コンテクスト: どのような状況で作るか・実装上の制約 (HOW)
  - 例: 言語、フレームワーク、バージョンなど

### 約束を守らない場合の対応方法

- 条件をより具体的に・細かく記載する
- 文字フォーマットで強調 (太字記号*をいれる)
- 制約を繰り返す (同じ制約文を複数回記述する)
- 別の言い方で同じ制約を記述する
