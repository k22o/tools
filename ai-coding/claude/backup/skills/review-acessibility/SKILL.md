# アクセシビリティレビュー

対象ファイルまたはディレクトリ `$1` のソースコードをアクセシビリティ観点で審査・修正する。

## ステップ 1 — 対象の特定

`$1` が指定されている場合はそのパスを対象とし、指定がなければ現在の git 差分（`git diff HEAD`）を対象とする。

対象ファイルの言語・フレームワークを確認する（HTML / JSX / TSX / Vue / Svelte / Angular テンプレートなど）。

## ステップ 2 — レビュー観点

以下の観点でコードを審査し、問題を列挙する。

### セマンティクスと構造

- 見出し階層（`<h1>`〜`<h6>`）が正しく入れ子になっているか
- ランドマーク要素（`<main>`, `<nav>`, `<header>`, `<footer>`, `<aside>`, `<section>`）が適切に配置されているか
- リスト要素（`<ul>`, `<ol>`, `<dl>`）が意味的に正しく使われているか
- ボタンとリンクが役割に応じて正しく使い分けられているか（遷移 → `<a>`、操作 → `<button>`）
- テーブルに `<caption>`, `<th scope>`, `<thead>` / `<tbody>` が揃っているか
- フォームの各入力に `<label for>` または `aria-label` / `aria-labelledby` があるか
- DOM の順序が視覚的な読み順と一致しているか（WCAG 1.3.2 / CSS で視覚位置だけ変えて DOM 順が逆になっていないか）
- `<div>`タグを乱用していないか、適切なタグを利用できているか

### HTML 構文の妥当性

- タグが正しく入れ子になっているか（例: `<p>` の中に `<div>`、`<a>` の中に `<button>` などの禁止入れ子がないか）
- `id` 属性がページ内で重複していないか
- 必須属性が欠落していないか（例: `<img>` の `alt`、`<input>` の `type`）
- 開始・終了タグの対応が正しいか（WCAG 4.1.1）

### ARIA とスクリーンリーダー対応

- `role` 属性が WAI-ARIA 仕様に沿って正しく使われているか（冗長な role の重複がないか）
- `aria-label`, `aria-labelledby`, `aria-describedby` が適切に設定されているか
- 動的コンテンツ変化を通知する `aria-live` リージョン（`polite` / `assertive`）が設定されているか
- トグル・展開系 UI に `aria-expanded`, `aria-pressed`, `aria-selected` が付与されているか
- `aria-hidden="true"` が装飾目的アイコン・重複テキストに正しく使われているか
- 画像に適切な代替テキストが設定されているか（WCAG 1.1.1）
  - 意味のある画像: 内容を説明する `alt` テキストがあるか
  - 装飾目的の画像: `alt=""` が明示的に設定されているか（属性自体の省略は不可）
  - SVG・アイコン: `aria-label` または `<title>` で代替テキストがあるか、装飾なら `aria-hidden="true"` になっているか
- エラーメッセージが `aria-errormessage` または `role="alert"` で通知されているか

### キーボード操作とフォーカス管理

- インタラクティブ要素がすべてキーボードで到達・操作できるか
- `tabIndex` の乱用がないか（`tabIndex > 0` はアンチパターン）
- モーダル・ドロワー・コンボボックスなどでフォーカストラップが正しく実装されているか
- モーダルクローズ後などフォーカスが適切な位置に戻るか
- フォーカスインジケーター（アウトライン）が CSS で非表示にされていないか（`outline: none` や `outline: 0` の乱用）
- `Skip to main content` リンクなどスキップナビゲーションが実装されているか（ページ全体を対象とする場合）
- キーボードショートカット（`accesskey`）が既存ショートカットと衝突していないか

### 色・コントラスト・視覚的提示

- 色だけで情報を伝えていないか（エラー表示が赤色のみ、など）
- 通常テキストのコントラスト比が 4.5:1 以上（WCAG AA）か（インラインスタイルや CSS 変数から読み取れる範囲で）
- 大テキスト（18px 以上 / 14px 太字以上）のコントラスト比が 3:1 以上か
- フォーカスインジケーターのコントラスト比が 3:1 以上か
- アニメーション・点滅要素に `prefers-reduced-motion` メディアクエリが考慮されているか
- テキストが画像化されていないか（代替テキストのない `<img>` でテキストが表示されていないか）

## ステップ 3 — 信頼スコアリング

問題ごとに 0〜100 の信頼スコアを付ける:

| スコア | 意味 |
|--------|------|
| 0 | 誤検知。根拠なし |
| 25 | 可能性あり。要確認 |
| 50 | 実際の問題だがマイナーまたは文脈依存 |
| 75 | 高確度の実問題。WCAG 基準に明確に抵触 |
| 100 | 確実。スクリーンリーダーや AT でユーザーがブロックされる |

**スコア 60 未満は除外する。**

## ステップ 4 — レポート出力と修正

### 出力フォーマット

```markdown
## アクセシビリティレビュー

### 🔴 Critical（スコア 90+）
N 件

1. **[WCAG 基準番号] 問題の概要**
   - 該当コード: `file.tsx:L42`
   - 問題: スクリーンリーダーがボタンの目的を読み上げられない
   - 修正案: `aria-label="メニューを開く"` を追加

### 🟠 High（スコア 75〜89）
...

### 🟡 Medium（スコア 60〜74）
...

---
参照基準: WCAG 2.1 AA / WAI-ARIA 1.2
```

### 自動修正

スコア **75 以上** かつ **機械的に修正可能な問題**（`alt` 属性の追加、`aria-label` の付与、`role` の修正など）は、直接ソースを修正する。  
修正内容はレポートの各項目に「✅ 修正済み」と記載する。

ロジックや UI デザインの変更を伴う問題（コントラスト比の調整など）は指摘のみとし、修正は行わない。

## 注意事項

- ビルド・型チェック・テスト実行は行わない（CI で別途実行される）
- WAI-ARIA の暗黙的ロール（例: `<button>` は `role="button"` を持つ）による重複指摘は false positive として除外する
- コンポーネントライブラリ（MUI, Radix UI など）が内部でアクセシビリティを担保している場合はそのコンポーネントの利用方法のみを審査する
- `aria-hidden` の配下にある要素の問題は指摘対象外
