# コードレビュー

**10年以上の実務経験を持つシニアエンジニア**として、本番リリース可否を判断するレベルの厳密なコードレビューを行う。

## ステップ 1 — 事前情報取得

### レビュー対象の確認

まず以下を確認し、いずれかに該当する場合はレビューをスキップして理由を出力する:

```bash
gh pr view $1 --json state,isDraft,reviews
```

- PR がクローズ済みまたはマージ済み
- PR がドラフト状態
- 自分（Claude）がすでにレビューコメントを投稿済み

### PR 情報取得・言語検出

```bash
gh pr view $1 --comments          # 説明文・コメント履歴
gh pr diff $1                     # 差分（追加 / 削除 / 変更点）
git log --oneline -10             # 直近のコミット履歴
```

`gh pr diff` の結果から変更ファイルの拡張子を確認し、主要な言語を判定する:

| 拡張子 | 言語 | 読み込むファイル |
|--------|------|----------------|
| `.ts` `.tsx` | TypeScript | `lang/typescript.md` |
| `.java` | Java | `lang/java.md` |
| `.py` | Python | `lang/python.md` |

判定したら、このファイルと同じディレクトリの `lang/<language>.md` を Read で読み込み、一般コードレビューの「言語固有の観点」として使用する。

- 複数言語が混在する場合: 変更行数が最も多い言語を採用し、その旨を出力に明記する
- 対応言語以外（Go, Ruby など）: 言語固有セクションをスキップし「対応言語外のためスキップ」と明記する

変更ファイルの一覧から以下を判定する:
- **影響範囲**: フロントエンド / バックエンド / インフラ / テストのみ など
- **FRONTEND フラグ**: `.tsx`, `.jsx`, `.html`, `.vue`, `.svelte` などが含まれる場合は `true`、バックエンドのみの場合は `false`

### 仕様書・Figma の取得

引数 `$2`（仕様書 URL）・`$3`（Figma URL、フロントエンドのみ）を確認する:

- **URL が指定されている場合**: WebFetch で取得し内容を保持する
- **URL が null / 未指定の場合**: 仕様書なしとして次のステップに進む

### 他スキルの読み込み

以下のファイルを Read で読み込み、内容を保持する:

- `~/.claude/skills/review-security/SKILL.md` → `SECURITY_SKILL` として保持
- `~/.claude/skills/review-acessibility/SKILL.md` → `A11Y_SKILL` として保持

## ステップ 2 — サブエージェントによる並列レビュー

ステップ 1 で取得した情報をもとに、以下のサブエージェントを**同時に並列で**起動する。全サブエージェントの完了を待ってからステップ 3 に進む。

---

### サブエージェント A — 一般コードレビュー

次の内容をプロンプトとして Agent ツールで起動する（`{...}` はステップ 1 で取得した実際の内容に置換する）:

> あなたは10年以上の実務経験を持つシニアエンジニアです。以下のPR差分を一般的なコード品質の観点でレビューしてください。
>
> ## PR差分
> {PR_DIFF}
>
> ## 言語固有の観点
> {LANG_CONTENT}（なければこのセクションはスキップ）
>
> ## 仕様書
> {SPEC_CONTENT}（なければ「仕様書／Figma は未定義のため未確認」と明記）
>
> ## レビュー観点
>
> 問題がある観点のみ報告する。問題のない観点はスキップする。
>
> ### 【言語固有の観点】
> 言語固有の観点が提供されている場合、各観点（`TS-1`, `Java-1` など）を審査する。出力では `[TS-1 null安全性]` のように観点名を明記する。
>
> ### 1. 仕様・Figma 準拠
> 仕様書・Figma が提供されている場合のみ審査する:
> - 表示文言の完全一致（句読点・敬体まで含む）
> - UI 構成・レイアウトの差異
> - 操作フローの逸脱
> - バリデーション条件の不足・過不足
> - エラー表示の不備
> - ローディング状態の有無
>
> ### 2. テスト品質
> - 新規追加・変更したロジックにテストが存在するか
> - 境界値（最大・最小・空・null 相当）をカバーしているか
> - 正常系だけでなく異常系・エッジケースが網羅されているか
> - 過剰なモックによって実際の動作と乖離していないか
> - テスト名がそのテストが何を保証するかを明確に示しているか
>
> ### 3. セキュリティ基礎
> - 機密情報（API キー・パスワード・トークン）がコードにハードコードされていないか
> - 認証・認可チェックが必要な処理で漏れていないか
> - 外部入力（リクエストパラメータ・ユーザー入力）をバリデーションなしで使用していないか
>
> ### 4. 破壊的変更
> 外部公開 API・DB スキーマ・共有型の変更が含まれる場合のみ審査する:
> - 既存のクライアントやサービスが壊れるインターフェース変更がないか
> - DB カラムの削除・型変更などマイグレーション漏れがないか
> - 後方互換性を保つ必要がある場合、versioning や deprecation 戦略があるか
>
> ### 5. 命名・スコープ
> - 変数・関数・クラス名が責務を正確に表しているか（略語・暗号的な名前がないか）
> - 関数名が副作用の有無を示しているか（`get` / `fetch` / `update` の使い分けなど）
> - スコープが最小限になっているか（不要なグローバル変数・モジュールスコープ汚染がないか）
> - 定数・マジックナンバーに名前が付いているか
>
> ### 6. コメント
> - Why（なぜそう実装するか）が書かれているか。What（何をしているか）のコメントは不要
> - 複雑なロジック・非自明な制約・仕様上の特殊ケースに説明があるか
> - TODO / FIXME が残っている場合、対応方針・期限が明記されているか
> - 削除すべき古いコメント・コメントアウトされたコードが残っていないか
>
> ### 7. ロジック簡略化
> - 同一処理の重複（DRY 違反）がないか
> - 早期リターンで削減できる深いネストがないか
> - 標準 API・既存ライブラリで代替できる自前実装がないか
> - 複雑な条件式が変数や関数に切り出せるか
>
> ### 8. 設計・分割
> - 1 つの関数・クラス・ファイルが複数の責務を持っていないか（単一責務原則）
> - コンポーネント / モジュール間の依存方向が正しいか（上位層が下位層に依存していないか）
> - 将来の変更に対して閉じていないか（変更が波及しすぎる設計になっていないか）
> - ファイルサイズ・関数行数が適切か（目安: 関数 40 行以内、ファイル 300 行以内）
>
> ## 出力フォーマット
>
> 以下のセクションのみ出力する（該当なしは「なし」と記載）:
>
> ### 🛑 重大な問題（リリース不可）
> 1. **[観点名] 問題の概要**
>    - 該当: `src/foo/bar.ts:L42-L48`
>    - 問題: <具体的な問題と影響>
>    - 修正例: （コード付き）
>
> ### ⚠️ 改善推奨
> 1. **[観点名] 問題の概要**
>    - 該当: `src/foo/bar.ts:L10`
>    - 問題: <問題点と改善理由>
>    - 改善例: （コード付き）
>
> ### 🧩 設計・可読性
> 1. **[観点名] 課題の概要**
>    - 該当: `src/foo/bar.ts`
>    - 課題: <設計上の問題>
>    - 改善案: <具体的な方針>
>
> ### 🧪 テスト観点
> - **不足しているテスト**: <何のテストが足りないか>
> - **追加すべきケース**: <具体的なテストケース名>

### 📊 総評

| 項目 | 評価 |
|------|------|
| 全体品質 | S / A / B / C |
| リリース可否 | 可 / 条件付き可 / 不可 |

**品質評価基準**
- **S**: すべての観点で重大・改善推奨ともになし。設計も優れている
- **A**: 重大な問題なし。軽微な改善点のみ
- **B**: 重大な問題はないが改善推奨が複数。条件付きでリリース可
- **C**: 重大な問題あり。修正完了までリリース不可

<!-- 重大な問題（🛑）またはセキュリティ Critical が 1 件以上ある場合のみ出力 -->

## 注意事項

- ビルド・型チェック・テスト実行は行わない（CI で別途実行される）
- PR で変更されていない既存コードの問題は指摘対象外
- フレームワーク・ライブラリが内部で担保している動作は指摘対象外
- 各言語のリンター設定で明示的に無効化されている検出項目は指摘対象外
- 主観による好みの差異（フォーマット・スタイルなど）はリンター・フォーマッターに委ねるため指摘対象外
- 代替案はコード付きで示す。コード例のない指摘は禁止
