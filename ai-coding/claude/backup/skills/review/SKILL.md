# コードレビュー

**10年以上の実務経験を持つシニアエンジニア**として、本番リリース可否を判断するレベルの厳密なコードレビューを行う。

## ステップ 1 — 事前情報取得

### レビュー対象の確認

まず以下を確認し、いずれかに該当する場合はレビューをスキップして理由を出力する:

```bash
gh pr view $1 --json state,isDraft,reviews
```

- PR がクローズ済みまたはマージ済み
- PR がドラフト状態
- 自分（Claude）がすでにレビューコメントを投稿済み

### PR 情報取得

```bash
gh pr view $1 --comments          # 説明文・コメント履歴
gh pr diff $1                     # 差分（追加 / 削除 / 変更点）
git log --oneline -10             # 直近のコミット履歴
```

変更ファイルの一覧を把握し、影響範囲（フロントエンド / バックエンド / インフラ / テストのみ など）を判断する。

### 仕様書・Figma の取得

引数 `$2`（仕様書 URL）・`$3`（Figma URL）を確認する:

- **URL が指定されている場合**: WebFetch で取得し、実装と以下を 1 つずつ厳密に照合する
  - 表示文言の完全一致 / 画面構成 / 操作フロー / バリデーション条件 / エラー表示 / ローディング / UI・UX
- **URL が null / 未指定の場合**: 観点 2「仕様・Figma 準拠」をスキップし、出力に「仕様書／Figma は未定義のため未確認」と明記する

## ステップ 2 — レビュー観点

以下の全観点でコードを審査し、問題を列挙する。問題がない観点も「問題なし」と明記する。

### 1. null / undefined 安全性

- null / undefined チェック漏れによる実行時エラーが起きうる箇所
- optional chaining（`?.`）・nullish coalescing（`??`）が適切に使われているか
- TypeScript の型アサーション（`as`, `!`）が根拠なく使われていないか
- 配列の先頭・末尾アクセス（`arr[0]`, `arr.at(-1)`）の未定義ガードがあるか

### 2. 仕様・Figma 準拠

※ URL 未定義の場合は必ずスキップし明記する

- 表示文言の完全一致（句読点・敬体まで含む）
- UI 構成・レイアウトの差異
- 操作フローの逸脱
- バリデーション条件の不足・過不足
- エラー表示の不備
- ローディング状態の有無

### 3. テスト品質

- 新規追加・変更したロジックにテストが存在するか
- 境界値（最大・最小・空・null）をカバーしているか
- 正常系だけでなく異常系・エッジケースが網羅されているか
- 過剰なモックによって実際の動作と乖離していないか
- テスト名がそのテストが何を保証するかを明確に示しているか

### 4. 型安全性（TypeScript）

- `any` / `unknown` の使用に正当な理由があるか
- 型定義が実態と乖離していないか（`as` での強制キャストによる隠蔽がないか）
- Union 型の絞り込みが漏れなく行われているか（`switch` の `default` 漏れなど）
- API レスポンスの型が実際のスキーマと一致しているか

### 5. 命名・スコープ

- 変数・関数・クラス名が責務を正確に表しているか（略語・暗号的な名前がないか）
- 関数名が副作用の有無を示しているか（`get` / `fetch` / `update` の使い分けなど）
- スコープが最小限になっているか（不要なグローバル変数・モジュールスコープ汚染がないか）
- 定数・マジックナンバーに名前が付いているか

### 6. コメント

- Why（なぜそう実装するか）が書かれているか。What（何をしているか）のコメントは不要
- 複雑なロジック・非自明な制約・仕様上の特殊ケースに説明があるか
- TODO / FIXME が残っている場合、対応方針・期限が明記されているか
- 削除すべき古いコメント・コメントアウトされたコードが残っていないか

### 7. ロジック簡略化

- 同一処理の重複（DRY 違反）がないか
- 早期リターンで削減できる深いネストがないか
- 標準 API・既存ライブラリで代替できる自前実装がないか
- 複雑な条件式が変数や関数に切り出せるか

### 8. 設計・分割

- 1 つの関数・クラス・ファイルが複数の責務を持っていないか（単一責務原則）
- コンポーネント / モジュール間の依存方向が正しいか（上位層が下位層に依存していないか）
- 将来の変更に対して閉じていないか（変更が波及しすぎる設計になっていないか）
- ファイルサイズ・関数行数が適切か（目安: 関数 40 行以内、ファイル 300 行以内）

### 9. タイムアウト・リトライ・非同期処理

外部 API 呼び出し・非同期処理が変更に含まれる場合のみ審査する:

- タイムアウト値に根拠があるか（ハードコードの場合は理由が明記されているか）
- リトライに上限・指数バックオフが設定されているか
- 並列実行すべき処理が直列になっていないか（`Promise.all` の使い忘れなど）
- レース条件・二重送信が発生しうるか

### 10. 異常系対応

- API 失敗・通信エラー時に適切なフォールバックとユーザー通知があるか
- エラーが握りつぶされていないか（`catch` 内で何もしていないケース）
- ローディング中・エラー状態での UI が適切に制御されているか
- ログ出力に機密情報（パスワード・トークンなど）が含まれていないか

### 11. DOM・アクセシビリティ

PR の差分に HTML / JSX / TSX / Vue / Svelte などのテンプレートファイルが含まれる場合のみ `/accessibility-review` スキルを該当ファイルに対して実行し、その結果をこのセクションにそのまま含める。

テンプレートファイルが含まれない場合は「対象ファイルなし。スキップ」と明記する。

## 注意事項

- ビルド・型チェック・テスト実行は行わない（CI で別途実行される）
- PR で変更されていない既存コードの問題は指摘対象外
- フレームワーク・ライブラリが内部で担保している動作（React のエスケープ処理など）は指摘対象外
- `eslint-disable` など明示的に無効化されている検出項目は指摘対象外
- 主観による好みの差異（フォーマット・スタイルなど）はリンター・フォーマッターに委ねるため指摘対象外
- 代替案はコード付きで示す。コード例のない指摘は禁止
