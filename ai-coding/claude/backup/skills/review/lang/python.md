# Python 固有の観点

### Python-1. None 安全性

- `AttributeError` / `TypeError` が発生しうる箇所（None チェック漏れ）がないか
- `Optional` な値（`dict.get()` の戻り値など）を直接使用していないか
- リストの先頭・末尾アクセス（`lst[0]`, `lst[-1]`）で空リストガードがあるか
- `None` を返す可能性のある関数の戻り値を検査せずに使っていないか
- ミュータブルなデフォルト引数（`def f(x=[])` / `def f(x={})` など）が使われていないか（呼び出し間で状態が共有される）

### Python-2. 型ヒント・静的解析

- 型ヒント（`typing` モジュール）が適切に付与されているか
- `Any` の使用に正当な理由があるか
- `Optional[X]` と `X | None`（Python 3.10+）の一貫した使い方になっているか
- `cast()` による強制型変換が根拠なく使われていないか
- mypy / pyright が通る型定義になっているか（`# type: ignore` の多用がないか）
- 戻り値の型アノテーションが省略されていないか

### Python-3. 非同期処理・リソース管理

非同期処理・外部リソースアクセスが変更に含まれる場合のみ審査する:

- `async/await` の使い忘れ（`await` なしで coroutine を使っていないか）
- 並列実行すべき処理が直列になっていないか（`asyncio.gather()` / `asyncio.wait()` の使い忘れ）
- ファイル・DB 接続・ソケットが `with` 文（コンテキストマネージャ）で確実にクローズされているか
- タイムアウト値に根拠があるか（`asyncio.wait_for` のタイムアウト設定など）
- レース条件・二重実行が発生しうるか
- `asyncio.run()` がトップレベル以外で呼ばれていないか

### Python-4. 例外・異常系対応

- 例外を `pass` や何もしない `except` ブロックで握りつぶしていないか（ログのみで再 `raise` しないケースも含む）
- `except Exception:` や裸の `except:` で広すぎる捕捉をしていないか（具体的な例外型を使う）
- `finally` 節でリソース解放が確実に行われているか（コンテキストマネージャが使えないケース）
- ログ出力に機密情報（パスワード・トークン・個人情報）が含まれていないか
- 例外メッセージがデバッグに十分な情報を含んでいるか
- カスタム例外が適切に定義・使用されているか

### Python-5. イミュータビリティ

- 再代入しない変数・定数に `Final`（`typing.Final`）を使っているか
- 変更されないコレクションに `list` ではなく `tuple` を使っているか
- 変更不要なデータクラスに `@dataclass(frozen=True)` を使っているか
- モジュールレベルの可変なコレクション（`list`, `dict`, `set`）がグローバル状態として意図せず変更されていないか

### Python-6. Docstring

- `public` な関数・クラス・メソッドに docstring があるか
- プロジェクト内で Google / NumPy / Sphinx スタイルが統一されているか
- 引数（`Args:`）・戻り値（`Returns:`）・例外（`Raises:`）が実態と一致して記述されているか
- 実装の「何をするか」ではなく「呼び出し側が知るべき前提条件・保証・副作用」が書かれているか
