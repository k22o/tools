# TypeScript 固有の観点

### TS-1. null / undefined 安全性

- null / undefined チェック漏れによる実行時エラーが起きうる箇所
- optional chaining（`?.`）・nullish coalescing（`??`）が適切に使われているか
- 型アサーション（`as`, `!`）が根拠なく使われていないか
- 配列の先頭・末尾アクセス（`arr[0]`, `arr.at(-1)`）の未定義ガードがあるか

### TS-2. 型安全性

- `any` / `unknown` の使用に正当な理由があるか
- 型定義が実態と乖離していないか（`as` での強制キャストによる隠蔽がないか）
- Union 型の絞り込みが漏れなく行われているか（`switch` の `default` 漏れなど）
- API レスポンスの型が実際のスキーマと一致しているか
- `noUncheckedIndexedAccess` 相当の配列アクセスが安全か
- ジェネリクスが適切に使われ、型情報が失われていないか

### TS-3. 非同期処理

外部 API 呼び出し・非同期処理が変更に含まれる場合のみ審査する:

- タイムアウト値に根拠があるか（ハードコードの場合は理由が明記されているか）
- リトライに上限・指数バックオフが設定されているか
- 並列実行すべき処理が直列になっていないか（`Promise.all` / `Promise.allSettled` の使い忘れなど）
- レース条件・二重送信が発生しうるか
- `async/await` と `.then()` が混在していないか
- unhandled Promise rejection が発生しうるか

### TS-4. 異常系対応

- API 失敗・通信エラー時に適切なフォールバックとユーザー通知があるか
- エラーが握りつぶされていないか（`catch` 内で何もしていないケース）
- ローディング中・エラー状態での UI が適切に制御されているか
- ログ出力に機密情報（パスワード・トークンなど）が含まれていないか

### TS-5. イミュータビリティ

- 再代入しない変数に `let` ではなく `const` を使っているか
- オブジェクト・配列のプロパティを変更しない場合に `readonly` / `Readonly<T>` / `ReadonlyArray<T>` を使っているか
- 関数の引数として受け取るオブジェクト・配列に `readonly` を付けて変更しない意図を明示しているか
- リテラルオブジェクト・配列を定数として扱う場合に `as const` を使っているか

### TS-6. DOM・アクセシビリティ

PR の差分に HTML / JSX / TSX / Vue / Svelte などのテンプレートファイルが含まれる場合のみ審査する:

- `innerHTML` / `dangerouslySetInnerHTML` が使われている場合、値がサニタイズ済みか確認する（XSS リスク）
- ユーザー入力を直接 DOM に挿入していないか
- アクセシビリティの問題は `/accessibility-review` スキルを該当ファイルに対して実行し、その結果をこのセクションにそのまま含める

テンプレートファイルが含まれない場合は「対象ファイルなし。スキップ」と明記する。
