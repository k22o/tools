# セキュリティレビュー

対象ファイルまたはディレクトリ `$1` のソースコードをセキュリティ観点で審査・修正する。

## ステップ 1 — 対象の特定

`$1` が指定されている場合はそのパスを対象とし、指定がなければ現在の git 差分（`git diff HEAD`）を対象とする。

対象ファイルの言語・フレームワーク・実行環境を確認する（Node.js / Python / Java / Go / ブラウザ向け JS/TS など）。

## ステップ 2 — レビュー観点

以下の観点でコードを審査し、問題を列挙する。

### インジェクション対策

- SQL クエリにユーザー入力を文字列結合で埋め込んでいないか（プレースホルダー・ORM を使っているか）
- HTML 出力時にユーザー入力をエスケープしているか（XSS 対策 / `innerHTML` への直接代入がないか）
- シェルコマンド実行にユーザー入力を渡していないか（コマンドインジェクション）
- ファイルパスにユーザー入力を使う場合、パストラバーサル（`../` など）を防いでいるか
- XML / HTML パーサーで外部エンティティ参照（XXE）が無効化されているか
- テンプレートエンジンでサーバーサイドテンプレートインジェクション（SSTI）が起こらないか

### 認証・セッション管理

- パスワードを平文または可逆な形式で保存していないか（bcrypt / argon2 など適切なハッシュを使っているか）
- セッション ID やトークンが推測困難な乱数で生成されているか
- JWT の署名アルゴリズムが `none` になっていないか、署名検証を省略していないか
- 認証済みチェックがすべての保護対象ルート・関数に適用されているか
- セッション固定攻撃を防ぐため、認証後にセッション ID を再生成しているか
- ログアウト時にサーバー側のセッションを無効化しているか

### 認可・アクセス制御

- ユーザーが自身のリソースにのみアクセスできるよう、所有者チェックをしているか（IDOR 対策）
- 権限チェックをクライアント側だけで行っていないか（必ずサーバー側でも検証しているか）
- 管理者・特権操作に対して追加の認可チェックがあるか
- 一括代入（Mass Assignment）によって意図しないフィールドが上書きされないか

### 機密情報の取り扱い

- API キー・パスワード・秘密鍵がソースコードにハードコードされていないか
- 機密情報がログ・エラーメッセージ・レスポンスボディに出力されていないか
- 環境変数や設定ファイルから機密情報を読み込む場合、デフォルト値に本番用シークレットを使っていないか
- クライアントに返却するオブジェクトに不要なフィールド（パスワードハッシュ・内部 ID など）が含まれていないか

### 暗号化・乱数

- MD5 / SHA-1 など脆弱なハッシュアルゴリズムをセキュリティ用途で使っていないか
- DES / RC4 など脆弱な暗号アルゴリズムを使っていないか
- セキュリティ用途の乱数に `Math.random()` など暗号論的に安全でないものを使っていないか（`crypto.randomBytes` / `secrets.token_bytes` などを使っているか）
- 暗号化に固定 IV / Nonce を使っていないか

### エラーハンドリング・情報漏洩

- スタックトレース・内部エラーメッセージをそのままクライアントに返していないか
- 例外をキャッチしているが何もしない（エラーの握りつぶし）になっていないか
- デバッグ用のコメント・ログ・エンドポイントが本番コードに残っていないか

### 依存関係

- `package.json` / `requirements.txt` / `build.gradle` などで既知の脆弱性を持つバージョンを指定していないか
- 不要または使われていない依存パッケージが含まれていないか

### HTTP・ブラウザセキュリティ（フロントエンド・サーバー含む）

- CORS 設定がオリジンを `*` に開放しすぎていないか
- Cookie に `HttpOnly` / `Secure` / `SameSite` 属性が付与されているか
- フォームや API に CSRF トークンが実装されているか（SameSite=Strict/Lax で代替している場合を除く）
- `eval()` / `new Function()` など動的コード評価を避けているか
- 以下のセキュリティ関連レスポンスヘッダーが設定されているか
  - `Content-Security-Policy`: スクリプト・スタイル・フレームなどの読み込み元を制限し XSS の影響を軽減する
  - `X-Content-Type-Options: nosniff`: ブラウザによる MIME タイプのスニッフィングを抑止し、意図しないコンテンツ解釈を防ぐ
  - `X-Frame-Options: DENY` または `SAMEORIGIN`: iframe 等への埋め込みを制限し、クリックジャッキングを防ぐ
  - `X-XSS-Protection: 1; mode=block`: ブラウザ組み込みの XSS フィルターを有効にし、検出時にページをブロックする
  - `Strict-Transport-Security`（HSTS）: HTTP ではなく HTTPS での通信を強制し、ダウングレード攻撃を防ぐ（例: `max-age=31536000; includeSubDomains`）

## ステップ 3 — 信頼スコアリング

問題ごとに 0〜100 の信頼スコアを付ける:

| スコア | 意味 |
|--------|------|
| 0 | 誤検知。根拠なし |
| 25 | 可能性あり。要確認 |
| 50 | 実際の問題だがマイナーまたは攻撃難易度が高い |
| 75 | 高確度の実問題。OWASP 基準に明確に抵触し悪用可能 |
| 100 | 確実。認証バイパス・RCE・機密情報漏洩など直接的な被害につながる |

**スコア 60 未満は除外する。**

## ステップ 4 — レポート出力と修正

### 出力フォーマット

```markdown
## セキュリティレビュー

### 🔴 Critical（スコア 90+）
N 件

1. **[CWE-XXX / OWASP A0X] 問題の概要**
   - 該当コード: `file.ts:L42`
   - 問題: ユーザー入力を直接 SQL クエリに結合しており SQLインジェクションが成立する
   - 修正案: プレースホルダー（`?` / `$1`）を使用する

### 🟠 High（スコア 75〜89）
...

### 🟡 Medium（スコア 60〜74）
...

---
参照基準: OWASP Top 10 2021 / CWE
```

### 自動修正

スコア **75 以上** かつ **機械的に修正可能な問題**（Cookie 属性の追加、`HttpOnly` 付与、ハードコードシークレットの環境変数参照への置き換えなど）は、直接ソースを修正する。  
修正内容はレポートの各項目に「✅ 修正済み」と記載する。

アーキテクチャ変更や外部設定が必要な問題（CORS 設定、CSP ポリシー定義、依存パッケージのアップグレードなど）は指摘のみとし、修正は行わない。

## 注意事項

- ビルド・テスト実行・依存関係の自動アップグレードは行わない（CI で別途実行される）
- 脆弱性があっても攻撃経路が存在しない（到達不能なコード・内部ネットワーク限定など）場合は false positive として除外する
- フレームワーク（Rails / Django / Spring など）が自動的に対策を提供している項目は、設定で無効化していない限り指摘対象外とする
- テストコード内のハードコード値（フィクスチャ用のダミーシークレットなど）は指摘対象外とする
