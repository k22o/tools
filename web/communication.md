# Web API 通信方式の比較

## REST API

### 概要
HTTP/1.1 をベースにした、リソース指向のアーキテクチャスタイル。

### 特性
- **リソース中心設計**: URL がリソースを表し、HTTP メソッド（GET / POST / PUT / DELETE）で操作を表現する
- **ステートレス**: サーバーはリクエスト間の状態を保持しない
- **キャッシュ可能**: HTTP のキャッシュ機構をそのまま利用できる
- **統一インターフェース**: 標準的な HTTP 規約に従うため、汎用クライアントで利用しやすい
- **テキストベース**: JSON / XML でやり取りするため人間が読みやすい

### 課題
- **Over-fetching**: 必要以上のデータが返ってくる
- **Under-fetching**: 1 回のリクエストで必要な情報が揃わず、複数回リクエストが必要になる（N+1 問題）
- **バージョン管理**: エンドポイントの変更に `/v1` `/v2` などのバージョニングが必要になりがち

### 向いているユースケース
- 公開 Web API・外部連携
- ブラウザクライアントから直接呼び出す場合
- リソース CRUD が中心のシンプルなサービス

## gRPC

### 概要
Google 製の RPC フレームワーク。HTTP/2 + Protocol Buffers（protobuf）を使用する。

### 特性
- **強い型付け**: `.proto` ファイルでスキーマを定義し、コードを自動生成する
- **バイナリ転送**: protobuf によるシリアライズで JSON より高速・軽量
- **HTTP/2 ベース**: ストリーミング（単方向・双方向）、多重化、ヘッダー圧縮が使える
- **4 つの通信パターン**:
  - Unary（1 リクエスト / 1 レスポンス）
  - Server Streaming
  - Client Streaming
  - Bidirectional Streaming
- **多言語対応**: protobuf から各言語のスタブを自動生成できる

### 課題
- ブラウザからの直接呼び出しが難しい（gRPC-Web が必要）
- バイナリ形式のためデバッグしにくい
- HTTP/1.1 ロードバランサーとの相性が悪いケースがある
- REST に比べてエコシステムが小さい

### 向いているユースケース
- マイクロサービス間の内部通信
- リアルタイム・ストリーミング処理
- 低レイテンシ・高スループットが求められる通信
- 多言語サービスが混在する環境

## GraphQL

### 概要
Facebook（Meta）製のクエリ言語 + ランタイム。クライアントが必要なデータ構造を宣言的に指定できる。

### 特性
- **クライアント主導のデータ取得**: 必要なフィールドだけをリクエストできる（Over/Under-fetching を解消）
- **単一エンドポイント**: `POST /graphql` の 1 エンドポイントにすべてのクエリが集約される
- **型システム**: SDL（Schema Definition Language）でスキーマを定義し、型安全なクエリを書ける
- **3 つの操作**:
  - `Query`（読み取り）
  - `Mutation`（書き込み・更新・削除）
  - `Subscription`（リアルタイム購読）
- **イントロスペクション**: スキーマ自体をクエリで取得でき、自己文書化されている

### 課題
- **N+1 問題**: リゾルバの実装次第で DB クエリが爆発する（DataLoader パターンで緩和）
- **キャッシュ**: REST のような HTTP キャッシュが使いにくい（クエリごとにキャッシュキーが異なる）
- **複雑なクエリ対策**: 深いネストや大量フィールドの悪意あるクエリへの防御が必要
- **学習コスト**: クエリ言語・リゾルバ設計の習得が必要

### 向いているユースケース
- フロントエンド（特に BFF: Backend for Frontend）
- 複数のクライアント（Web・モバイル）が異なるデータ形状を要求する場合
- 複雑なドメインモデルを持つ API

## 比較表

| 項目 | REST | gRPC | GraphQL |
|---|---|---|---|
| プロトコル | HTTP/1.1（HTTP/2 も可） | HTTP/2 | HTTP/1.1 / HTTP/2 |
| データ形式 | JSON / XML | Protocol Buffers（バイナリ） | JSON |
| スキーマ定義 | OpenAPI（任意） | `.proto`（必須） | SDL（必須） |
| 型安全性 | 低〜中 | 高 | 高 |
| パフォーマンス | 中 | 高 | 中 |
| ストリーミング | 限定的 | 双方向対応 | Subscription で対応 |
| ブラウザ対応 | ◎ | △（gRPC-Web 必要） | ◎ |
| Over/Under-fetching | 発生しやすい | 発生する | 解消できる |
| キャッシュ | HTTP キャッシュが使える | 独自実装が必要 | 工夫が必要 |
| 学習コスト | 低 | 中 | 中〜高 |
| 主な用途 | 外部公開 API | マイクロサービス内部通信 | BFF・フロントエンド向け |
